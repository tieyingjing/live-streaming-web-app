## 详细说明

### 为什么浏览器不能推流？

```
┌─────────────────────────────────┐
│  浏览器的三大限制                │
├─────────────────────────────────┤
│ 1. 不支持 RTMP 协议              │
│    ├─ RTMP 使用 TCP 1935 端口    │
│    └─ 浏览器只能使用 HTTP/WS     │
│                                 │
│ 2. Flash 插件已淘汰              │
│    ├─ RTMP 原本依赖 Flash        │
│    └─ 2020年底 Flash 停止支持    │
│                                 │
│ 3. 安全策略限制                  │
│    └─ 浏览器无法访问任意 TCP 端口 │
└─────────────────────────────────┘
```

## 为什么要区分推流和观看？

### 推流 (Upload/上传) - 需要专业工具

```
推流 = 把实时音视频编码后上传到服务器

要求:
├─ 视频编码 (H.264)
├─ 音频编码 (AAC)
├─ RTMP 协议支持
├─ 低延迟传输
└─ 稳定的上传带宽

难度: ⭐⭐⭐⭐⭐ (高)

浏览器: ❌ 能力不足
专业工具: ✅ OBS/Larix 等
```

### 观看 (Download/下载) - 浏览器完全够用

```
观看 = 从服务器下载视频流并播放

要求:
├─ HLS 协议支持 (HTTP)
├─ 视频解码 (硬件加速)
└─ 自适应码率

难度: ⭐⭐ (低)

浏览器: ✅ 完美支持
```

## 未来可能支持的推流方式

### WebRTC 浏览器推流 (阶段 5 计划)

```
手机/电脑浏览器 → WebRTC → 服务器 → 转 RTMP

优势:
✅ 无需安装 App
✅ 打开网页即可推流
✅ 超低延迟 (< 1秒)

劣势:
❌ 需要 WebRTC 服务器
❌ 开发成本高
❌ 服务器成本高

状态: 📅 计划中
```

**技术架构**:
```
浏览器 getUserMedia()
    ↓
WebRTC RTCPeerConnection
    ↓
WebRTC 媒体服务器 (Janus/Mediasoup)
    ↓
FFmpeg 转换 (WebRTC → RTMP)
    ↓
你的 RTMP 服务器 (已有)
```

## 常见问题

### Q: 为什么不直接支持浏览器推流？

**A**:
1. RTMP 协议限制 - 浏览器无法使用
2. 实现成本高 - 需要 WebRTC 额外架构
3. 已有成熟方案 - OBS 和 Larix 等工具完善

---

### Q: 手机浏览器真的不能推流吗？

**A**:
- ❌ RTMP 推流: 完全不能
- ✅ WebRTC 推流: 技术上可行，但需要额外开发
- ✅ 观看直播: 完全支持

---

### Q: 我只有手机，怎么推流？

**A**: 下载推流 App
```
iOS:
└─ App Store → Larix Broadcaster (免费)

Android:
└─ Google Play → Larix Broadcaster (免费)

配置:
├─ URL: rtmp://你的服务器IP:1935/live
└─ Key: stream_key
```

## 推荐配置

### Desktop 推流 (OBS)

```
视频:
├─ 分辨率: 1920x1080 或 1280x720
├─ 帧率: 30 fps
├─ 编码器: x264
├─ 码率: 2500 kbps (720p) / 4500 kbps (1080p)
└─ 预设: veryfast

音频:
├─ 码率: 128 kbps
└─ 采样率: 44.1 kHz

推流:
├─ 服务器: rtmp://localhost:1935/live
└─ 密钥: stream_key
```

```

音频:
├─ 码率: 128 kbps
└─ AAC 编码

推流:
├─ URL: rtmp://你的IP:1935/live
└─ Key: stream_key
```
